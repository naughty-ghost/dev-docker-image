# CLAUDE.md

このドキュメントは、Claude Codeがこのリポジトリで作業する際の**振る舞い・プロセス・規約**を定義します。
詳細なルールは `.claude/rules/` 配下のカテゴリ別ファイルに分割されています。

**トレードオフ**: これらのガイドラインは速度よりも慎重さを優先します。

## 1. 作業前の計画

**推測しない。混乱を隠さない。メリット、デメリットを明確にする。**

実装する前に:

- コードを修正する前に、必ず対象ファイルと関連ファイルを読み、理解してください。調査していないコード、情報を推測しないで下さい。
- コードのスタイル、慣習、および抽象化を厳密かつ徹底的に確認してください。
- 自身の仮定を明確に述べてください。不明確な場合は質問してください。
- 複数の解釈が存在する場合は、それらを提示してください。黙って一つを選ばないでください。
- よりシンプルな解決策がある場合は、それを提示してください。正当な理由がある場合は、反対意見を述べてください。
- 不明な点がある場合は、立ち止まってください。何が混乱を招いているかを特定し、質問してください。

## 2. シンプルさの優先

**問題を解決する最小限のコード。**

- 要求された以上の機能は追加しない。
- 1回限りのコードのために抽象化を行わない。
- 要求されていない柔軟性や将来の可能性を盛り込まない。
- 起こりえないシナリオに対するエラーハンドリングは行わない。
- 50行で済むものを200行で書いた場合は書き直してください。
- 最小限の変更かどうか自問自答してください。もしyesなら簡素化してください。

## 3. 不要な変更を行わない

**必要な箇所のみ触れる。自分が作成したゴミを片付ける。**

- 解決策は既存のコードスタイル、慣習に従ってください。
- 隣接するコード、コメント、またはフォーマットを「改善」のために変更しないでください。
- 自身の変更によって未使用となったインポート、変数、定数、関数を削除してください。

## 4. 成功基準まで反復する

**成功基準を定義する。基準を満たすまで反復する。**

タスクを検証可能な目標に変換してください：

- 「検証を追加」→「無効な入力に対するテストを書き、それをパスさせる」
- 「バグを修正」→「バグを再現するテストを書き、それをパスさせる」
- 「Xをリファクタリング」→「前後のテストがパスすることを確認する」

## 5. ドキュメント言語

**ドキュメントはすべて日本語で書く**

- プロジェクト内のドキュメントはすべて日本語で書いてください。
- 応答はすべて日本語で行います。
